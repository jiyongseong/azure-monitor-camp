# 모니터링이 어려운 이유

> 아래 내용은 [AutomationEdge – Application_Monitoring.pdf](https://rpa.automationedge.com/hubfs/Collateral/Inforgraphic/Application_Monitoring.pdf) 글을 정리하였습니다.

## 어렵게 하는 것들

모니터링이 어려운 가장 큰 이유는, 우리가 보고 싶은 건 "문제가 있는지"가 아니라 "사용자에게 어떤 영향이 생겼는지"인데 그 연결고리를 만들기가 쉽지 않기 때문입니다. 

### 1. End User Experience(최종 사용자 경험)에 대한 인사이트 부족 
첫 번째 과제로 **End User Experience(최종 사용자 경험)에 대한 인사이트 부족**을 꼽습니다. 사용자가 실제로 앱을 어떻게 쓰고 있는지, 어디서 불편을 겪는지에 대한 '행동 기반 정보'가 부족하면, 성능 지표가 정상처럼 보이더라도 체감 문제를 놓치기 쉽습니다. 

### 2. 데이터 과부하
두 번째는 **데이터 과부하(Data Overload)**입니다. 모니터링 시스템은 방대한 데이터를 만들어내는데, 그 안에서 의미 있는 신호를 골라내는 일이 점점 더 어려워집니다. 결국 "데이터는 많은데, 결론은 안 보이는" 상황이 생기고, 팀은 노이즈를 걸러내느라 시간을 쓰게 됩니다.

### 3. 지역·디바이스 커버리지의 한계
세 번째는 서비스가 사용하는 환경이 다양해지면서 생기는 **지역·디바이스 커버리지의 한계**입니다. 사용자는 서로 다른 네트워크/지역/단말에서 접속하고, 이 차이 때문에 특정 환경에서만 발생하는 문제는 재현도 어렵고 원인 파악도 늦어집니다.

### 4. 앱은 끊임없이 변한다
또 하나의 현실은 **"앱이 계속 변한다"** 는 점입니다. 통상 애플리케이션이 진화하는 동안 모니터링 설정과 계측(Instrumentation)을 최신으로 유지하는 것 자체가 어렵다고 합니다. 시스템이 바뀌는데 모니터링이 그대로면, 중요한 지표는 빠지고 의미 없는 지표만 남는 불일치가 생기게 됩니다.

### 5. 장애 사전 탐지
마지막으로, **사전 예방적 탐지(Proactive issue detection)의 어려움**이 있습니다. 장애가 터진 뒤에 알게 되는 것이 아니라, 터지기 전에 감지하려면 더 많은 데이터를 실시간으로 확인해야 하고, 그 운영 부담이 커집니다. 

## 그래서 
모니터링은 단순히 '도구를 하나 붙이는 일'로 끝나지 않습니다. **가시성(Visibility)을 확보하고 → 인사이트(Insights)를 얻고 → 최적화(Optimization)와 대응(Respond)으로 이어지는 흐름**이 필요합니다. 

Azure Monitor는 바로 이 흐름을 운영 관점에서 제공하는 플랫폼으로 이해하면 쉽습니다. 다양한 데이터 소스(애플리케이션/인프라/Azure 리소스/커스텀 소스)에서 신호를 모으고, 수집·라우팅·변환을 표준화하며, 핵심 데이터 플랫폼(Metrics와 Logs)을 중심으로 **시각화(Visualize), 분석(Analyze), 대응(Respond)** 까지 연결합니다.

이제부터는 이 "시각화" 영역에 집중해서, 같은 데이터를 두고도 왜 Azure Dashboard / Azure Workbooks / Azure Managed Grafana / Power BI가 각각 다른 선택이 되는지, 그리고 어떤 상황에서 무엇을 고르면 좋은지 초보자 관점에서 차근차근 풀어보겠습니다.
